# [OPEN] tileset-camera-stutter

## 背景
- 问题：模型体积较小，但在放大模型后移动镜头查看细节时仍出现卡顿。
- 目标：定位运行时瓶颈，优化 3DTiles 渲染与镜头交互流畅度，并基于证据输出优化结论。

## 当前状态
- 状态：已创建调试会话，待补充运行时证据。
- 约束：在拿到运行时证据前，不修改业务逻辑；第一处逻辑改动仅允许加入埋点。

## 初始假设
1. 镜头移动时触发过于激进的瓦片请求与卸载，导致主线程和 GPU 资源调度抖动。
2. 当前动态屏幕误差、中心优先加载、跳层加载等参数组合不适合近距离观察，造成细节阶段频繁重算与切换。
3. 模型放大后可见瓦片数量骤增，但内存预算或缓存命中不足，出现反复解码与上传 GPU 资源。
4. Cesium 场景在每帧都完整渲染，缺少请求渲染模式约束，镜头交互期的帧耗时放大。
5. 并非 3DTiles 本身，而是场景中的其他渲染/事件链路在相机变化时带来额外负担。

## 计划
1. 识别 3DTiles 加载与相机更新代码路径。
2. 仅加入调试埋点，记录帧耗时、相机变化、瓦片可见数量、请求数量、内存/误差关键指标。
3. 复现并收集日志。
4. 根据证据收敛根因后再做最小优化。
5. 对比优化前后数据并整理结论。

## 埋点设计
- A：场景 `postRender` 低频采样，记录近似 FPS、慢帧占比、相机是否移动、相机高度与变化量。
- B：`add3DTileset()` 入口记录本次加载参数，确认当前 LOD/裁剪/动态误差配置。
- C：`tileset.loadProgress` / `allTilesLoaded` 记录请求堆积、处理中瓦片数量与 tileset 统计。
- E：`Viewer` 初始化时记录场景渲染模式，确认是否存在持续重绘。

## 当前复现要求
- runId：`pre-fix`
- 日志文件：`.dbg/trae-debug-log-tileset-camera-stutter.ndjson`
- 复现路径：
  1. 打开首页并加载一个会发生卡顿的 `tileset.json` 模型。
  2. 将镜头尽量贴近模型表面或边角细节。
  3. 保持放大状态，连续平移、旋转、俯仰镜头 10-15 秒。
  4. 如果出现明显卡顿、掉帧或纹理/细节切换抖动，即视为复现成功。

## 证据分析
| ID | 假设 | 状态 | 证据摘要 |
|----|------|------|----------|
| A | 镜头移动引发调度抖动 | ✅ 已确认 | 日志显示移动近景时 FPS 从 80-120 掉到 10-45，慢帧占比最高达到 1。 |
| B | 当前 LOD 参数不适合近景移动 | ✅ 已确认 | 当前近景仍使用 `maximumScreenSpaceError=12`，且叶子优先、飞行预加载、渐进分辨率同时开启。 |
| C | 缓存/显存溢出导致瓦片反复装载 | ✅ 已确认 | `totalMemoryUsageInBytes` 多次升到 0.9GB-1.1GB，远高于 `cacheBytes=512MB`，同时 `pendingRequests` 最多到 7、`tilesProcessing` 最多到 24。 |
| D | 持续重绘是主因 | ❌ 否决为主因 | 静止时 FPS 可回到 108-120，说明常态帧循环不是主要瓶颈。 |
| E | 其他业务链路抢占主线程 | ⏳ 暂无直接证据 | 日志中卡顿与 tileset 请求/内存/命令数抬升高度同步，暂未看到其他链路先于卡顿放大。 |

## 已实施修复
1. 将日志 runId 切换为 `post-fix`，保留原埋点用于修复后对比。
2. 将 3DTiles 缓存预算改为显式 `cacheBytes` / `maximumCacheOverflowBytes`，避免近景移动时缓存过度溢出。
3. 将 `maximumScreenSpaceError` 调整为更稳妥的默认值，并在镜头移动时临时放宽、停止后延迟恢复。
4. 关闭 `preferLeaves`、`preloadFlightDestinations`、`progressiveResolutionHeightFraction`，减少近景移动时的额外请求风暴。
5. 关闭小模型近景场景下不必要的 `dynamicScreenSpaceError`，收敛 LOD 波动。
